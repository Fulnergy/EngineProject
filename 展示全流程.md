## 三、PDF 明确要求的 QA（必问，答错该 Task 直接 0 分）

### Task 4 必问（共 3 题）

**Q1: Explain how to design snapshot in begin command.**

BEGIN 时调用 `createSnapshot()`：先将当前内存状态持久化到磁盘（`persistRuntimeState` → BufferPool.FlushAllPages + MetaManager.saveToJson + DiskManager.dump_disk_manager_meta），然后用 `Files.walk` 把整个数据库目录复制到一个 `Files.createTempDirectory("cs307-txn-")` 临时目录。这个临时目录副本就是快照。回滚时用它覆盖原目录。

**Q2: Explain if executing commit command, what happens in physical and logical structure.**

物理层：`commit()` 调用 `persistRuntimeState()` — BufferPool 中所有脏页 Flush 到磁盘，MetaManager 的 JSON 写入磁盘，DiskManager 的文件元数据 JSON 写入磁盘。保证数据持久化。然后 `cleanupTransaction` 删除 BEGIN 快照和所有 SAVEPOINT 快照目录。

逻辑层：`inTransaction` 置为 false，SAVEPOINT 栈清空。后续 SQL 在事务外执行，SAVEPOINT 操作会抛 `TRANSACTION_REQUIRED` 异常。

**Q3: Explain your design of savepoint and rollback.**

SAVEPOINT：调用 `createSnapshot()` 创建当前数据库目录的完整副本，存入 `LinkedHashMap<String, Deque<Path>>`。同名保存点用栈管理 — 后创建的遮蔽先创建的。ROLLBACK TO SAVEPOINT：从栈顶 peek（不 pop）获取快照，`restoreFromSnapshot` 清空当前目录并复制快照回来，然后 `reloadInMemoryState` 刷新 MetaManager（重新从 JSON 加载）和 BufferPool（ClearAll 清空缓存）。保存点保留以便重复使用。RELEASE SAVEPOINT：pop 栈顶并删除对应快照目录。

### Task 1 必问

**Q4: LRU 和 Clock 的设计区别？什么场景用哪个？**

LRU：LinkedList（头 = LRU 最旧，尾 = MRU 最新）+ HashSet O(1) 查找。每次命中 O(n) 移动节点到尾部，淘汰 O(1) 取头部。精确但访问开销大。

Clock：环形数组 + refBit。每次命中 O(1) 设 refBit=1。淘汰 O(n) 转圈：refBit=1 → 改为 0 给第二次机会；refBit=0 → 淘汰。近似但访问极快。

真实数据库（PostgreSQL、MySQL）倾向 Clock 变体：淘汰稍不精准但每次访问几乎零开销，整体性能更好。

**Q5: Pin/Unpin 在 BufferPool 中的作用？生命周期？**

FetchPage 时 pin_count++，如果从 0→1 调用 `Pin(frameId)` 标记为不可淘汰。上层用完 `unpin_page` 使 pin_count--，降到 0 时调用 `Unpin(frameId)` 放回 Replacer 成为可淘汰。Pin/Unpin 保证正在使用的页面不会被错误淘汰。

脏页（dirty page）：内存中修改过但未写回磁盘的页。淘汰脏页前必须先 FlushPage 写回，否则数据丢失。

### Task 3 必问

**Q6: B+树内部节点和叶子节点的结构？分裂过程？**

内部节点：`List<Value> keys` + `List<Node> children`，只存 key + 子节点指针，用于路由。

叶子节点：`List<Value> keys` + `List<RID> values` + `LeafNode next`，存所有数据 + 指向下一个叶子的链表指针。

分裂：叶子 key 数 > maxKeys → 拆成两半，右半第一个 key 作为分隔 key 提升到父节点。父节点溢出 → 递归分裂。根分裂 → 创建新根，B+树高度+1（树长高的唯一方式）。

**Q7: CREATE INDEX 内部做了什么？IndexScan 如何工作？**

CREATE INDEX：1. MetaManager 记录索引元数据到 JSON；2. 创建空 B+树；3. SeqScan 全表扫描，逐行将该列的值和 RID 插入 B+树；4. B+树存入 DBManager.indexes Map。

IndexScan：Begin() 从 B+树全量取出所有 RID，按 pageNum 排序减少随机 I/O。Next() 通过 RID 直接 GetRecord 读取记录。比 SeqScan 逐 slot 扫描 bitmap 效率高。

**Q8: B+树 vs B 树的核心区别？为什么数据库都用 B+树？**

B+树数据只存叶子节点，内部节点只存索引。叶子有链表，范围查询 O(log N + K)。B 树数据存在所有节点，范围查询需要中序遍历更慢。B+树还有一个优势：内部节点不存数据，同样大小的节点能存更多 key，树更矮，IO 次数更少。

### Task 2 必问（队友需要准备）

**Q9: Explain SeqScan implementation and execution logic.**

SeqScanOperator 实现火山模型。Begin()：OpenFile 打开 .db 文件，读 FileHeader 获取总页数和每页记录数。hasNext()：从当前 page+slot 扫描 bitmap，找到 isSet 的 slot 返回 true。Next()：通过 RID(pageNum, slotNum) 调用 GetRecord 读磁盘，slot++。Current()：Record 包装为 TableTuple 返回。Close()：CloseFile 释放文件句柄。

**Q10: Explain COUNT operation design.**

COUNT 通过 AggregateOperator 实现。Begin() 初始化 count=0。hasNext() 调用下层算子逐行拉取，每拉一行 count++。Current() 返回 count。如果有 WHERE 条件，下层 FilterOperator 先过滤再计数。上层判断 isCountQuery → 创建 LogicalAggregateOperator 替代 LogicalProjectOperator。
