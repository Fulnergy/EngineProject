### Task 3：B+树索引性能对比

> PDF 要求 "Use large enough datasets for testing" + "Ensure correctness and performance of index functionality"

**Step 1：无索引 — SeqScan**

```sql
-- 查看执行计划：走 TableScan → SeqScan
explain select * from big where big.score = 99;

-- 执行查询（逐页逐 slot 扫描 500 行）
select * from big where big.score = 99;
```

**Step 2：创建索引 + 打印 B+ 树**

```sql
create index idx_score on big(score);
-- 1. MetaManager JSON 记录索引元数据
-- 2. SeqScan 全表扫描 → 逐行 (score, RID) 插入 B+ 树
-- 3. B+ 树叶子溢出 → 分裂 → key 提升 → 树增高
-- 4. B+ 树存入 DBManager.indexes

-- 通过代码调用 BPlusTree.printTree() 打印节点结构
```

**Step 3：有索引 — IndexScan**

```sql
-- 执行计划变为 BPlusTreeIndexScan
explain select * from big where big.score = 99;

-- 同一查询：B+ 树 O(log N) 定位 → 按 RID 精确读磁盘
select * from big where big.score = 99;
```

**Step 4：范围查询全走索引**

```sql
select * from big where big.score = 85;       -- 等值
select * from big where big.score > 90;       -- 大于
select * from big where big.score < 30;       -- 小于
select * from big where big.score > 50 and big.score < 60;  -- 区间
```

**Step 5：多索引 + 清理**

```sql
create index idx_id on big(id);
-- 不同列走不同索引
select * from big where big.score = 99;       -- idx_score
select * from big where big.id = 250;         -- idx_id

drop index idx_score;
drop index idx_id;
explain select * from big where big.score = 99;  -- 回到 SeqScan
```

事项: 现场改 B+树节点数
说明: 老师可能说"把 order 从 4 改成 8"。你去 DBManager.java:261 改 new BPlusTree(4) → new BPlusTree(8)，重新跑 createindex，树会变矮（内部节点容量翻倍）
**对比总结**


|            | 无索引 (SeqScan)        | 有索引 (IndexScan)                |
| ---------- | ----------------------- | --------------------------------- |
| 查找方式   | 逐页逐 slot 扫描 bitmap | B+ 树二分定位 → 叶子链表         |
| 时间复杂度 | O(N) 500 行全扫         | O(log N) ≈ 2-3 次节点跳转        |
| 物理算子   | `SeqScanOperator`       | `BPlusTreeIndexScanOperator`      |
| EXPLAIN    | `TableScan → SeqScan`  | `TableScan → BPlusTreeIndexScan` |

### Task 4：事务

```sql
create table users (id int, name char);

begin;
insert into users (id, name) values (1, 'Alice');
savepoint sp1;
insert into users (id, name) values (2, 'Bob');
savepoint sp2;
insert into users (id, name) values (3, 'Charlie');
select * from users;
-- 3 行

rollback to savepoint sp2;
select * from users;
-- 2 行

release savepoint sp2;
rollback to savepoint sp1;
select * from users;
-- 1 行

commit;
select * from users;
-- 持久化为 1 行

rollback;
-- 事务外空操作，不报错
```

## 三、注意事项

### Task 4

**Q1: Explain how to design snapshot in begin command.**

BEGIN 时调用 `createSnapshot()`：先将内存状态持久化到磁盘（`persistRuntimeState` → BufferPool.FlushAllPages + MetaManager.saveToJson + DiskManager.dump_disk_manager_meta），然后用 `Files.walk` 把整个数据库目录复制到 `Files.createTempDirectory("cs307-txn-")` 临时目录。这个副本就是快照。

**Q2: Explain if executing commit command, what happens in physical and logical structure.**

物理层：`commit()` → `persistRuntimeState()` 把脏页 Flush、MetaManager JSON 和 DiskManager meta JSON 写磁盘。`cleanupTransaction` 删除 BEGIN 和所有 SAVEPOINT 快照目录。

逻辑层：`inTransaction = false`，SAVEPOINT 栈清空。后续 SQL 不在事务中，SAVEPOINT 操作抛 `TRANSACTION_REQUIRED`。

**Q3: Explain your design of savepoint and rollback.**

SAVEPOINT：`createSnapshot()` 创建目录副本，存入 `LinkedHashMap<String, Deque<Path>>`（栈语义：同名后创建遮蔽先创建）。ROLLBACK TO：`stack.peek()` 获取快照 → `restoreFromSnapshot` 覆盖目录 → `reloadInMemoryState` 刷新 MetaManager + BufferPool。保存点不 pop（可复用）。RELEASE：`stack.pop()` + 删快照目录。

### Task 3

**Q4: B+树内部节点和叶子节点结构？分裂过程？**

内部节点：`keys + children`，只存路由。叶子节点：`keys + values(RID) + next 链表`，存所有数据。分裂：key 数 > maxKeys → 拆两半 → 右半最小 key 提升到父节点 → 递归 → 根分裂则树高度+1。

**Q5: CREATE INDEX 内部过程？IndexScan 工作原理？**

CREATE INDEX：元数据记录 JSON → 建空 B+树 → SeqScan 全表逐行插入 → 存入 DBManager.indexes。IndexScan：Begin() 从 B+树取全量 RID 排序 → Next() 按 RID 精确 GetRecord。

**Q6: B+树 vs B 树？为什么数据库用 B+树？**

B+树数据只在叶子，内部节点只存索引，叶子有链表；范围查询 O(log N + K)。B 树数据散落各层，范围需中序遍历。B+树叶子的扇出更大，树更矮，IO 更少。
