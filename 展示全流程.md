## 三、PDF 必问 QA（答错该 Task 直接 0 分）

### Task 4 必问（共 3 题）

**Q1: Explain how to design snapshot in begin command.**

BEGIN 时调用 `createSnapshot()`：先将内存状态持久化到磁盘（`persistRuntimeState` → BufferPool.FlushAllPages + MetaManager.saveToJson + DiskManager.dump_disk_manager_meta），然后用 `Files.walk` 把整个数据库目录复制到 `Files.createTempDirectory("cs307-txn-")` 临时目录。这个副本就是快照。

**Q2: Explain if executing commit command, what happens in physical and logical structure.**

物理层：`commit()` → `persistRuntimeState()` 把脏页 Flush、MetaManager JSON 和 DiskManager meta JSON 写磁盘。`cleanupTransaction` 删除 BEGIN 和所有 SAVEPOINT 快照目录。

逻辑层：`inTransaction = false`，SAVEPOINT 栈清空。后续 SQL 不在事务中，SAVEPOINT 操作抛 `TRANSACTION_REQUIRED`。

**Q3: Explain your design of savepoint and rollback.**

SAVEPOINT：`createSnapshot()` 创建目录副本，存入 `LinkedHashMap<String, Deque<Path>>`（栈语义：同名后创建遮蔽先创建）。ROLLBACK TO：`stack.peek()` 获取快照 → `restoreFromSnapshot` 覆盖目录 → `reloadInMemoryState` 刷新 MetaManager + BufferPool。保存点不 pop（可复用）。RELEASE：`stack.pop()` + 删快照目录。

### Task 1 必问

**Q4: LRU 和 Clock 的设计区别？**

LRU：LinkedList（头=LRU 最旧，尾=MRU 最新）+ HashSet。命中 O(n) 移动节点，淘汰 O(1)。精确但开销大。

Clock：环形数组 + refBit。命中 O(1) 设 refBit=1，淘汰 O(n) 转圈给第二次机会。近似但访问极快。PostgreSQL/MySQL 倾向 Clock 变体。

**Q5: Pin/Unpin 的作用？脏页是什么？**

FetchPage 时 pin_count 从0→1 调 `Pin(frameId)` 保护。`unpin_page` 将 pin_count 降到0 时调 `Unpin` 放回 Replacer。脏页=内存修改过未写回磁盘的页，淘汰前必须 FlushPage。

### Task 3 必问

**Q6: B+树内部节点和叶子节点结构？分裂过程？**

内部节点：`keys + children`，只存路由。叶子节点：`keys + values(RID) + next 链表`，存所有数据。分裂：key 数 > maxKeys → 拆两半 → 右半最小 key 提升到父节点 → 递归 → 根分裂则树高度+1。

**Q7: CREATE INDEX 内部过程？IndexScan 工作原理？**

CREATE INDEX：元数据记录 JSON → 建空 B+树 → SeqScan 全表逐行插入 → 存入 DBManager.indexes。IndexScan：Begin() 从 B+树取全量 RID 排序 → Next() 按 RID 精确 GetRecord。

**Q8: B+树 vs B 树？为什么数据库用 B+树？**

B+树数据只在叶子，内部节点只存索引，叶子有链表；范围查询 O(log N + K)。B 树数据散落各层，范围需中序遍历。B+树叶子的扇出更大，树更矮，IO 更少。

### Task 2 必问（队友）

**Q9: SeqScan 火山模型实现？**

Begin()：OpenFile → 读 FileHeader（总页数+每页记录数）。hasNext()：扫描当前 page+slot 的 bitmap，遇到 isSet 返回 true。Next()：RID → GetRecord → slot++。Current()：包装为 TableTuple。Close()：CloseFile。

**Q10: COUNT 怎么实现的？**

AggregateOperator。Begin() 初始化 count=0。hasNext() 从下层逐行拉取（可能经 FilterOperator 过滤），每拉一行 count++。Current() 返回 count。isCountQuery 判断 → LogicalAggregateOperator 替代 LogicalProjectOperator。
