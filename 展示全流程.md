### Task 3：B+树索引性能对比

**Step 1：无索引 — SeqScan**

```sql
explain select * from big where big.score = 99;
select * from big where big.score = 99;
```

**Step 2：建索引 + 打印 B+树结构**

```sql
create index idx_score on big(score);
-- 自动打印 B+树 (createIndex 里调了 tree.printTree()):
-- Depth 0: [60]           ← 根节点
-- Depth 1: [23, 43] [87]  ← 内部节点
-- ...
-- Leaf chain: LeafNode[...]→next  ← 叶子链表
```

**Step 3：有索引 — IndexScan**

```sql
explain select * from big where big.score = 99;
select * from big where big.score = 99;
```

**Step 4：范围查询**

```sql
select * from big where big.score = 85;
select * from big where big.score > 90;
select * from big where big.score < 30;
select * from big where big.score > 50 and big.score < 60;
```

**Step 5：多索引 + 清理**

```sql
create index idx_id on big(id);
select * from big where big.score = 99;  -- idx_score
select * from big where big.id = 250;    -- idx_id

drop index idx_score;
drop index idx_id;
explain select * from big where big.score = 99;  -- 回到 SeqScan
```

**现场改节点数**：改 B+树 order。 `DBManager.java:261`，`new BPlusTree(4)` → `new BPlusTree(8)`，重建索引即可。

**对比**


|            | 无索引 (SeqScan)      | 有索引 (IndexScan)         |
| ---------- | --------------------- | -------------------------- |
| 查找方式   | 逐页逐 slot 扫 bitmap | B+树二分定位 → 叶子链表   |
| 时间复杂度 | O(N) 全扫             | O(log N)                   |
| 物理算子   | SeqScanOperator       | BPlusTreeIndexScanOperator |

### Task 4：事务

```sql
create table users (id int, name char);

begin;
insert into users (id, name) values (1, 'Alice');
savepoint sp1;
insert into users (id, name) values (2, 'Bob');
savepoint sp2;
insert into users (id, name) values (3, 'Charlie');
select * from users;                    -- 3 行

rollback to savepoint sp2;
select * from users;                    -- 2 行

release savepoint sp2;
rollback to savepoint sp1;
select * from users;                    -- 1 行

commit;
select * from users;                    -- 持久化 1 行

rollback;                               -- 事务外空操作，不报错
```

## 三、注意事项

### Task 4

**Q8: Explain how to design snapshot in begin command.**

BEGIN 调用 `createSnapshot()`：先 `persistRuntimeState` 把脏页/MetaManager JSON/DiskManager JSON 写磁盘，再 `Files.walk` 复制整个数据库目录到临时目录。快照就是这套副本。

**Q9: Explain if executing commit command, what happens.**

物理层：`persistRuntimeState()` 持久化，`cleanupTransaction` 删所有快照。逻辑层：`inTransaction=false`，SAVEPOINT 栈清空，事务外操作抛 `TRANSACTION_REQUIRED`。

**Q10: Explain your design of savepoint and rollback.**

SAVEPOINT：`createSnapshot()` 创建目录副本，存入 `LinkedHashMap<String, Deque<Path>>`（栈语义）。ROLLBACK TO：`stack.peek()` → `restoreFromSnapshot` 覆盖目录 → `reloadInMemoryState` 刷新 MetaManager + BufferPool，保存点不 pop。RELEASE：`stack.pop()` + 删快照。

### Task 3

**Q11: B+树内部节点和叶子节点结构？分裂过程？**

内部节点：`keys + children`，只存路由。叶子：`keys + RID + next`，存数据 + 链表。分裂：key 超 maxKeys → 拆两半 → 右半最小 key 提升到父 → 递归 → 根分裂树高+1。

**Q12: CREATE INDEX 做了什么？IndexScan 如何工作？**

CREATE INDEX：MetaManager 记录 JSON → 建空 B+树 → SeqScan 全表逐行插入 → 存入 DBManager.indexes。IndexScan：Begin() 从 B+树取全量 RID 排序 → Next() 按 RID 精确 GetRecord。

**Q13: B+树 vs B 树？为什么数据库用 B+树？**

B+树数据只在叶子，叶子有链表，范围 O(log N+K)。B 树数据散落各层，范围需中序遍历。B+叶子扇出更大，树更矮，IO 更少。
